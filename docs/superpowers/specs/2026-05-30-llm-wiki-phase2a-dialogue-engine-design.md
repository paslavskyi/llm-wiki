## 1. Мета і скоуп

Побудувати **двигун діалогу** — спосіб, у який LLM веде з користувачем структуровану розмову, що збирає знання та систематизує їх у mind-map поверх атомарних нотаток Phase 1.

Це **підсистема 2a** з трьох (рухаємось покроково, одна за раз):
- **2a — Двигун діалогу (цей spec):** скіл `kb-elicit` + мінімальний mind-map субстрат.
- **2b — Еволюція й цілісність (пізніше):** `kb-evolve`, `impact.mjs`, `graph.mjs`, `health.md`, pre-commit.
- **2c — Безперервність (пізніше):** `kb-recap`, `session-delta.mjs`, `journal/`.

**Поза скоупом 2a:** мульти-нотаткова еволюція з ripple (2b), журнал/самері сесій (2c), повноцінний `autonomous`-режим (двигун працює в `debug` — власник у циклі), генерація артефактів `docs/` (Phase 3).

## 2. Принципи двигуна

1. **Спочатку карта, потім деталі.** Не збираємо подробиці, доки не домовились про *області*, які треба зібрати. Карта — теж знання.
2. **Відкритий, перевикористовуваний фреймворк.** Таксономія областей не зашита; вона співтвориться під кожну базу знань. Цей репо — прототип для інших баз.
3. **Зверху вниз (mind-map).** Область → під-області → конкретні нотатки. Розгортається рекурсивно.
4. **Тригер збереження — логічна консистентність.** Нотатку зберігаємо, коли вона стала цілісною, завершеною думкою. Двигун ставить уточнювальні питання, щоб *добудувати* цілісність перед записом.
5. **Невизначеність не губиться.** Невирішене питання стає нотаткою (`Q-`/`ASMP-`/`RISK-`), а не зникає.
6. **Адаптивно + трекінг покриття.** Дозволяємо стрибати за енергією розмови, але двигун веде агенду й підсвічує прогалини.
7. **Проактивна експертиза.** Двигун пропонує структуру, кидає виклик припущенням, на запит досліджує (web/докази) і повертає висновок у нотатки.

## 3. Метод діалогу

### Етап 0 — спільне створення карти (framing)
Запускається, коли база порожня або користувач хоче переглянути карту.
1. Двигун пропонує **сід** областей верхнього рівня (для продуктової бази — vision, market, users, product, roadmap, gtm; як рекомендований кістяк, не догма).
2. Разом адаптуємо/доповнюємо: додаємо, перейменовуємо, прибираємо області під конкретну базу («ідеї», «критерії», «технічні вимоги» — як назве користувач).
3. Кожна узгоджена область верхнього рівня фіксується як `topic`-нотатка з `parent: null` → їй відповідає папка.
4. Результат Етапу 0 — верхній рівень mind-map. `STATE.md` оновлюється: фаза «карту узгоджено».

### Етап 1 — занурення зверху вниз (drill)
Для обраної теми-вузла:
1. **Каркас.** Двигун пропонує «що тут варто зібрати» (під-вузли/аспекти) і узгоджує його → створює дочірні `topic`-вузли (`parent: <topic>`).
2. **Наповнення.** По кожному під-вузлу веде розмову, ставлячи уточнювальні питання, **доки нотатка не стане логічно консистентною**, тоді зберігає її через `kb-capture` з полем `topic: <вузол>`.
3. **Відкриті місця** фіксує як `Q-`/`ASMP-`/`RISK-` нотатки, підвішені до того ж вузла.
4. Рекурсивно йде глибше або повертається до агенди.

### Наскрізні правила
- **resolve перед записом** (через `kb-recall`) — не плодимо дублі (правило Phase 1 `kb-capture`).
- **Підтвердження** потрібне тільки коли запис *переписує* наявне знання; нове знання фіксуємо щойно воно консистентне (у `debug` — показуємо що записали).
- **Дослідження на запит:** користувач може попросити «запропонуй» чи «дослідь» — двигун виконує і повертає результат у нотатки (з джерелами, якщо це research).

## 4. Mind-map субстрат (файли та граф)

**Принцип:** папки — пласкі «коробки» верхнього рівня (області); ієрархія mind-map живе у графі (`parent`/`topic`), а не в глибині папок. Уникаємо churn при переродженні гілок.

- Вузол карти = нотатка `type: topic`, id `TOP-`.
- Область верхнього рівня = `topic` із `parent: null`; їй відповідає папка `knowledge/<area>/`.
- Під-вузли = `topic` із `parent: TOP-xxx`; лежать **пласко** в папці своєї області.
- Конкретна нотатка несе `topic: TOP-xxx` — вузол, до якого вона підвішена.
- Ієрархія — лише через frontmatter. `domainOf` (Phase 1) і далі працює: домен = папка = область.

**Приклад:**
```
knowledge/
  vision/                        # область = TOP-001
    TOP-001-vision.md            #   type: topic, parent: null
    TOP-007-problem-space.md     #   type: topic, parent: TOP-001
    VIS-001-mission.md           #   concrete, topic: TOP-001
    PAIN-003-manual-entry.md     #   concrete, topic: TOP-007
  users/
    TOP-002-users.md             #   parent: null
    TOP-010-primary-persona.md   #   parent: TOP-002
    PER-001-busy-parent.md       #   concrete, topic: TOP-010
index/
  MAP.md            # домени/лічильники (є)
  mindmap.md        # ГЕНЕРУЄТЬСЯ: дерево областей→під-областей→нотаток
  <area>.index.md   # (є)
  backlinks.json    # (є)
```

Phase 1 папки — це **сід** для цієї бази; Етап 0 їх переглядає. Структура приймає будь-який набір областей.

## 5. Зміни тулінгу (тестоване ядро 2a)

1. **`tools/schema/topic.json`** — нова схема: `type` enum `["topic"]`; `id` патерн `^TOP-[0-9]+$`; обовʼязкові базові поля (`id,type,title,status,summary`); `parent` — `string` (патерн `^TOP-[0-9]+$`) або `null`.
2. **`tools/schema/registry.json`** — додати `"topic": "topic.json"`.
3. **`tools/schema/base.json`** — додати опційне поле `topic` (`type: string`). (`parent` лишається лише в topic-схемі.)
4. **`tools/validate.mjs`** — розширити перевірку резолюції посилань: значення `parent` (на topic-нотатках) і `topic` (на конкретних нотатках), коли присутні, мають резолвитись в наявну нотатку, **тип якої = `topic`**. Інакше — помилка (`unresolved topic/parent → …` або `parent/topic must point to a topic note`).
5. **`tools/reindex.mjs`** — генерувати `index/mindmap.md`: з `topic`-нотаток будуємо дерево за `parent` (корені — `parent:null`); під кожним вузлом — конкретні нотатки, у яких `topic` = id вузла; рендер вкладеним Markdown-списком із банером GENERATED. Дерево детерміноване (сортування за id). Нотатки без `topic` групуються в секцію «(unassigned)».

Усе через TDD з тимчасовими фікстурами, як у Phase 1.

## 6. Скіл `kb-elicit` (двигун)

Новий скіл `.claude/skills/kb-elicit/SKILL.md`. Прозова реалізація методу з §3.

**Що робить:** веде структуровану розмову — Етап 0 (карта) або Етап 1 (занурення) — і перетворює діалог на нотатки.

**Кроки (скорочено):**
1. Прочитати `kb.config.yml` (режим/мова) і зорієнтуватись (`kb-orient`): чи є карта (`index/mindmap.md` / `topic`-нотатки)?
2. **Нема карти →** Етап 0: запропонувати сід областей, узгодити, створити `topic` верхнього рівня через `kb-capture`.
3. **Є карта →** показати агенду (порожні області з MAP + відкриті `Q-` + вузли без наповнення) і запропонувати, куди занурюватись; узгодити вузол.
4. Для вузла: узгодити каркас (під-вузли), потім по кожному вести розмову до логічної консистентності → зберігати через `kb-capture` (`topic: <вузол>`); відкриті місця → `Q-`/`ASMP-`/`RISK-`.
5. На запит — пропонувати/досліджувати; результати фіксувати в нотатки.
6. Оновити `STATE.md` (поточний вузол, наступний крок).

**Поведінка за режимом:**
- `debug`: показувати, які нотатки створено/оновлено, ID, агенду.
- `autonomous`: короткі підтвердження (повна полірувка — пізніше; 2a орієнтований на `debug`).

**Залежності:** `kb-orient`, `kb-recall` (resolve), `kb-capture` (запис). Не залежить від 2b/2c.

## 7. Карта покриття / агенда

Без окремого артефакту, що дрейфує. Агенда = композиція трьох джерел:
1. **Порожні області** — з `index/MAP.md` (лічильники = 0).
2. **Відкриті `Q-`/`ASMP-`/`RISK-`** — явні відомі прогалини.
3. **Вузли mind-map без наповнення** — `topic` без дочірніх конкретних нотаток (з `index/mindmap.md`).

`kb-elicit` зводить це у «що далі». `STATE.md` лишається коротким знімком (поточний вузол + рекомендований крок), не дублем агенди.

---

## 8. Зміни в існуючих файлах

- **`CLAUDE.md`:** додати тип `topic`/`TOP-` у словник; згадати `kb-elicit` у протоколі сесії; згадати `index/mindmap.md` як генерований; описати поля `parent`/`topic`.
- **`STATE.md`:** відобразити, що двигун діалогу доступний; фаза = збір знань.
- **Словник типів:** `topic` додається до продуктового/крос-секційного набору (область/під-область — структурна, не доменно-специфічна).

---

## 9. Тестова стратегія

- **Юніт (Node, `node:test`):** topic-схема валідна/невалідна; `validate` ловить нерезолвлений `parent`/`topic` і `parent`/`topic`, що вказує не на `topic`; `reindex` будує коректне дерево `mindmap.md` (вкладеність за `parent`, конкретні нотатки під своїм вузлом, секція unassigned, детермінованість).
- **Інтеграційний smoke:** створити маленьку карту (2 області, 1 під-вузол, 2 конкретні нотатки) → validate чисто → `mindmap.md` має очікуване дерево → прибрати.
- **Скіл** (`kb-elicit`): перевірка валідності frontmatter; ручний прохід методу — не юніт-тест.
